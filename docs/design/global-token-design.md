## 问题背景

对图表图片进行等比例放大/缩小的大规模测试中发现，模型在不同分辨率下的正确率呈**倒 U 形**：中间分辨率表现好，两端（极端高分辨率、极端低分辨率）表现差。

**当前阶段目标**：修复高分辨率端的性能下降。低分辨率端暂时观察，可能随高分辨率端的改动一并改善。

## Q1 — 为什么要加全局 token？

**结论**：高分辨率下局部 token 缺少整体感受野，图表的整体结构（坐标轴、图例、趋势线）被分散到大量 token 中，模型难以聚合出全局语义。加入全局 token 是第一步修复。

## Q3 — 为什么选双线性下采样而不是 embedding 空间聚合？

**结论**：选择在**像素空间**做双线性下采样（`F.interpolate(mode='bilinear')`）。

理由：全局 token 的目标是**模拟原始（中等）分辨率下的观感**。图片被放大后，每个 32×32 区域对应更小的物理区域；将其下采样回 16×16，相当于还原到原始分辨率下该区域的样子。这是在像素空间而非 embedding 空间做聚合的核心动机——保留原图倍率下的低频视觉结构，而不是 4 个局部 embedding 的数学均值。

备选方案（未选）：在 embedding 空间取 4 个局部 token 均值，更轻量但缺少像素级低频信息。

## Q7 — DeepStack 注入层是否解冻？

**结论**：解冻 `deepstack_merger_list`，LLM 本体保持冻结。

背景：Qwen3-VL 的 DeepStack 机制在 ViT 第 8、16、24 层提取视觉特征，经 `deepstack_merger_list`（线性投影）注入 LLM 对应层。

理由：fold 操作使 DeepStack 提取点的 token 值变为 `x_i + α·x5`，merger 输入分布实质已改变。让 merger 跟着视觉编码器一起更新，相当于让它充当"新视觉分布 → LLM 期望空间"的适配器，LLM 作为固定优化目标，merger 学习满足该目标。

|阶段	| 解冻 |	备注 |
|--------|------|------|
|Stage 1（warm-up） |	global_proj + α × 4	global | 通路先建立，其余冻结 |
S|tage 2（联合） |	视觉编码器 + deepstack_merger_list	| 一起解冻联合优化 |

- **LLM 本体**（attention、MLP 等）：保持冻结
- **`deepstack_merger_list`**：解冻，与视觉编码器在 Stage 2 同步训练

## Q14 — ViT QKV 权重是否需要针对 x5 特殊处理？

**结论**：不需要任何特殊处理，QKV 权重形状不变，Stage 2 正常 SFT 即可。

**关键认知**：ViT 的 QKV 投影是**全序列共享**的线性变换（`Q = W_Q · x`），不存在"针对第 5 个 token 的专用权重"。W_Q/W_K/W_V 对 x1-x4 和 x5 使用同一套矩阵，**权重形状完全不变，无需扩展，无需插 0**。

**真实需要适配的**：预训练时 W_Q/W_K/W_V 只见过 x1-x4 风格的局部 token。引入 x5（全局 patch）后，attention 需要学会：
- x5 如何 query/key/value 局部 token
- 局部 token 如何 attend to x5

这是权重**数值**适配，不是形状问题。Stage 2 解冻视觉编码器（含所有 ViT block 的 QKV）时，梯度自然流过，正常 SFT 完成适配。

**初始扰动小的原因**：`global_proj` 复制 `proj` 权重，x5 embedding 初始分布与 x1-x4 相近，W_Q * x5 ≈ W_Q * x_local，训练起点合理。

## Q9 — 折叠后全局 token 是否在后续 ViT 段中继续参与？

### 主方案（当前选择）：x5 跨段保留并重新插入

每次 DeepStack 提取前折叠时，**保存 x5 的值**；下一段开始前将保存的 x5 重新插入序列，使全局 token 贯穿整个 ViT：

```
Layer 0–8：   [x1, x2, x3, x4, x5]  → 5N/4 token
折叠（保存 x5_8）：x_i = x_i + x5，删除 x5
Layer 8–16：  重新插入 x5_8 → [x1', x2', x3', x4', x5_8]  → 5N/4 token
折叠（保存 x5_16）：x_i = x_i + x5，删除 x5
Layer 16–24：  重新插入 x5_16 → [x1'', x2'', x3'', x4'', x5_16]  → 5N/4 token
折叠（保存 x5_24）：最终折叠
输出：N token
```

全局 token 在每一段都参与 attention，信息最充分。

### 备选方案 Y-lite（待尝试）：仅前 8 层为 5N/4

- Layer 0–8：5N/4 token，x5 参与 attention
- 第一次折叠后 x5 不再恢复
- Layer 8 之后：N token，全局 token 消失
- 计算更省，效果上限较低，可作为消融对比

## Q10 — 重新插入 x5 时的位置编码与 cu_seqlens 处理

**结论**：采用推荐方案——cu_seqlens 和 position_embeddings 全程统一使用 5N/4 版本，进入 ViT 前计算一次，不再变化。折叠与重新插入只修改 `hidden_states` 的值，不改变序列长度，因此 Flash Attention 2 的边界信息始终正确。

重新插入的 x5 沿用最初计算的 rotary PE（取 4 个局部 token 位置均值），无需在每段重新计算。

⚠️ **保留问题**：重新插入时 x5 使用的是上一段处理后的语义值，但 PE 是原始值——跨段的语义与位置是否存在不一致，待实验验证。

## Q11 — 折叠门控系数的粒度

**结论**：采用**每折叠点一个可学习标量 α**，共 4 个（3 次 DeepStack 折叠 + 1 次最终折叠），全部初始化为 **0.01**。

折叠公式：`x_i = x_i + α_k * x5`，其中 k 为当前折叠点编号（0–3）。

初始化为 0.01（小正数）而非 0：保证 fold 路径梯度从训练开始就能流动（α=0 时 `∂L/∂global_proj` 经 fold 路径为 0），同时初始影响极小，模型从接近预训练基线的状态出发。

**直觉**：ViT 不同深度对全局信息的需求不同——浅层（0–8）处理低级纹理，深层（16–24）处理语义结构，各阶段的最优融合比例应由数据决定，而非手工设定。

### 四种粒度方案对比（均已记录，供后续参考）

| 方案 | 参数量 | 优点 | 缺点 | 状态 |
|------|--------|------|------|------|
| 全局一个 α | 1 | 最简单 | 无法区分不同处理阶段 | 备选 |
| **每折叠点一个 α** | 4 | 区分各处理阶段，参数极少 | — | ✅ 当前选择 |
| 每空间组一个 α | N/4 | 空间位置级别控制 | 参数多，易过拟合 | 备选 |
| 每局部位置各一个（x1–x4） | 4×组数 | 区分左上/右下融合比例 | 复杂度高，必要性不明确 | 备选 |

## Q12 — global_proj 初始化 与 α=0 梯度问题

### 核心矛盾

"不影响预训练行为"与"梯度能流动"是对立的：初始化越接近 0，预训练行为保留越好，但梯度死得越干净。

### α=0 的梯度问题（已确认）

fold 路径：`x_i = x_i + α * x5`

当 α=0 时，`∂L/∂global_proj` 经 fold 路径传回的梯度 = 0，`global_proj` 在训练初期完全无法通过 fold 路径学习。

### 关于"QKV 插入权重 0"

ViT 的 Q、K、V 投影是全序列共享的线性变换，不存在"第 5 个位置专用权重"。插入 x5 不会造成 x1-x4 的 QKV 矩阵错位；真正的风险是 x5 通过 attention 内积影响 x1-x4 的输出。这无法通过在 attention 权重里插 0 来解决。

### 补充：QKV 错位问题的澄清

此 ViT 使用**空间 (h,w) RoPE**（见 `rot_pos_emb`），相对位置基于图像坐标而非 tensor index。插入 x5 不改变 xi1 与 xj1-xj4 之间的相对空间位置，**RoPE 层面无硬性错位**。

真实存在的软性影响：softmax 分母增大，x5 分走部分 attention 权重（软性稀释），模型 fine-tuning 中自适应。

### 结论（双尺度视角修正）：`global_proj` 复制 `proj` + α=0.01

在"双尺度叠加"视角下，x5 是一个合法的原始分辨率 16×16 patch，格式与 x1-x4 完全相同，**复制 `proj` 比随机初始化更语义合理**。x5 通过 attention 与 x1-x4 的交互是有意为之，正是双尺度设计的目的。

| 组件 | 初始化 | 理由 |
|------|--------|------|
| `global_proj` | 直接复制 `proj` | 语义合理；x5 从"合法 patch embedding"出发；attention 交互有意为之 |
| α | 0.01（小正数） | fold 路径梯度可流；α 自行从小值增长 |

各方案对比：
- `global_proj=0, α=0`：梯度完全死，什么都学不了
- `global_proj=随机小值, α=0.01`：梯度可流，但语义起点无意义
- `global_proj=复制proj, α=0`：语义合理，但 fold 路径梯度为 0
- ✅ `global_proj=复制proj, α=0.01`：语义合理 + 两条路径梯度均可流

## Q8 — 折叠操作的时机（核心架构决策）

### 折叠策略

ViT 处理完毕后，将全局 token（x5）加回 4 个局部 token，**恢复原始序列长度**，使 Merger 及下游所有层无需改动：

```
ViT 处理中：[x1, x2, x3, x4, x5]  →  5N/4 个 token，x5 参与 attention
折叠后：
    x1 = x1 + x5
    x2 = x2 + x5
    x3 = x3 + x5
    x4 = x4 + x5
    删除 x5
结果：[x1', x2', x3', x4']  →  N 个 token，恢复原始长度
```

### 方案选择

**当前选择：方案 Y（每次 DeepStack 提取前折叠 + 最终输出前折叠）**

- 在第 8、16、24 层 DeepStack 提取前各执行一次折叠
- 所有 ViT block 结束后再执行一次最终折叠
- **优点**：DeepStack merger 完全不需要改动，merger 也不需要改动，下游无感知
- **缺点**：block 循环中需要加钩子，折叠逻辑分散在多处

**备选方案：方案 X（所有 block 结束后统一折叠）**

- 仅在所有 ViT block 结束后执行一次折叠
- **优点**：逻辑最简单，折叠只写一次
- **缺点**：DeepStack 提取时仍面对 5N/4 个 token，deepstack merger 的形状扩展仍然需要
- **状态**：暂不实现，后续可能作为对比实验尝试
